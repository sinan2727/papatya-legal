## 1.5 Age Limit

Papatyam serves only users over the age of 18. During registration, an age confirmation step is presented: the user declares via a checkbox that they are over 18 and enters their 4-digit year of birth (day/month is not requested — in accordance with the data minimization principle of KVKK Art.4). Only three pieces of information are written to the server: `yasOnayi` (boolean), `dogumYili` (year as an integer), and `yasOnayTarihi` (server timestamp, for proof of explicit consent).

The age declaration is self-attested (the user's responsibility); it is technically possible for a malicious user to circumvent it, but legal responsibility lies entirely with the user. Accounts later found to belong to someone under 18 are immediately suspended and the user data is purged in cascade through the standard account deletion flow.

When the age-limit clause of our policy is updated, users who registered under an older version are asked to re-confirm their age from within the application (similarly to the KVKK re-consent chain).

## 3. Architecture: Zero-Knowledge Approach

The most important feature of Papatyam is that the server does **not see the cleartext form** of user data. This is the fundamental design principle of the application, and it is implemented through the following technical measures:

- **The phone number is never stored in cleartext.** The number is hashed on your device using the HMAC-SHA256 algorithm with a pepper key (a server-side secret key distributed via Firebase Remote Config and rotated with a short TTL). The server knows only this hash; it has no access to the actual phone number.
- **The PIN is not sent to the server.** On your device, the PIN is transformed into a master key using the PBKDF2-HMAC-SHA256 algorithm with 100,000 iterations. The master key is kept in your device's memory and is not transmitted to the server.
- **E-mail, profile information, and past messages are stored encrypted.** They are encrypted using the AES-256-CBC algorithm with the master key (or the device UUID). The server sees only the encrypted form of this data; it cannot decrypt it.
- **Free messages are end-to-end encrypted.** The cryptography used: X25519 ECDH key exchange, per-message AES-GCM encryption (12-byte nonce + integrity tag), and TOFU (Trust On First Use) pinning of the other party's public key. Our server stores the encrypted content (ciphertext) but cannot decrypt the plaintext — this is why we say "we normally cannot read message contents." This is not a Signal-absolute guarantee: template messages (50 predefined text templates + 12 emoji-only templates) are processed unencrypted for the multi-recipient flow and the need for anonymous moderation; this is a deliberate trade-off so that user complaints can be evaluated.

This architecture ensures that, in the event of any data breach, the information an attacker could obtain is limited.

## 4. Data Collected and Its Purposes

### 4.1 Data We Collect Directly

- **Phone number (hashed):** Only its HMAC-SHA256 + pepper hashed form. Used to authenticate the user's identity and to enable contact matching. The cleartext form is never present on the server.
- **E-mail address (encrypted, optional):** For account recovery and match notifications. Stored encrypted with the master key or UUID.
- **PIN (hashed on device):** Hashed with PBKDF2-HMAC-SHA256. The hash is stored on the server for verification; the raw PIN value is not accessible to anyone.
- **Backup code (hashed on device):** A 6-digit code used in case of PIN loss. Stored as a hash.
- **FCM push token:** The device token generated by Firebase to send notifications. It is a temporary identifier of the device, not tied to the user's identity.
- **Profile information (optional, encrypted):** Gender, age range, city, interests. Entirely under the user's control, not mandatory to fill in, and can be changed or deleted at any time.
- **Settings:** Contact access preference, notification channel, PIN prompt frequency, language preference.
- **Explicit consent record:** Proof of the KVKK explicit consent you gave during onboarding (see 4.5).
- **Age confirmation record:** The declaration you gave for the age gate + year of birth (year only) + confirmation date. Details in Section 1.5.

### 4.2 Contact Access

The application also hashes the phone numbers of the contacts in your address book with HMAC-SHA256 + pepper to perform match checking. **The cleartext form of your contacts (names, numbers) is never sent to the server.** Only the hashed phone numbers are compared against the other hashes on the server for the purpose of match checking.

### 4.3 Automatically Collected Data

- **Application usage counters:** Like counter, template message counter, game counter (for monthly limit management).
- **Crash/error reports (Firebase Crashlytics):** When performing crash/error reporting with Crashlytics, we establish a pseudonymous user link. A `user_uid_hash` of 8 hex characters derived via HMAC from your account UID, and a `premium_status` flag (premium / free), are written to Crashlytics. Your raw UID, phone number, or any other identifying information is **never** sent to Crashlytics. The `user_uid_hash` is one-way (reversal is practically impossible) and does not repeat across different accounts. This allows us to correlate multiple crashes of the same user to determine root cause; we do not learn your identity. Crashlytics also receives non-PII technical information such as application version, build type (debug/release), platform (Android/iOS), and device language.
- **Rate limit records:** The number of hourly/daily requests per user, for spam protection. Pseudonymous.

### 4.4 Data We Do Not Collect

- Your cleartext phone number (only the hash)
- The names of your contacts (only phone hashes)
- The cleartext of your free messages (only end-to-end encrypted ciphertext)
- Your location information (the application does not request location permission)
- Advertising identifiers (there are no ads in the application)
- Cookies, usage analytics (there is no tracking such as Google Analytics or Firebase Analytics)

### 4.5 Explicit Consent (KVKK Art.5/1, GDPR Art.6/1/a)

During onboarding, after phone verification, a checkbox is presented in which you confirm that you have read the Privacy Policy and give your explicit consent to the processing of your personal data within the scope of this policy. Registration cannot be completed without checking this box. When you give consent, the following three pieces of information are written to our server:

- `kvkkOnayTarihi` — server timestamp (for evidentiary purposes, the burden of proof under KVKK Art.5/1).
- `kvkkPolitikaSurumu` — the version of the policy you accepted (in this document, "1.1").
- `kvkkPolitikaURL` — the publication address of the policy at that moment (`https://sinan2727.github.io/papatya-legal/privacy_policy`).

When the policy version is updated, users who registered under an older version are asked from within the application to renew their explicit consent; until consent is renewed, changed features may be restricted.

### 4.6 Use of Artificial Intelligence (AI)

In Papatyam, artificial intelligence is used for only a single purpose: to generate generic, open-ended conversation questions in the Secret Hunt (Sır Avı) section (for example, "What was the moment that made you laugh the most in your life?").

- **None of your user data is sent to the AI.** Your messages, your identity, your contacts, your profile, or any information belonging to the other party is not transmitted to the AI system.
- The AI does not analyze or profile you or the other party; it makes no predictions such as age, gender, or intent.
- The generated question is first shown to you; no question is sent without you seeing and approving it. You can edit the question, regenerate it, or not use it at all.
- The Anthropic Claude API is used as infrastructure (see Section 7); only a general instruction for question generation is sent to this service, and none of your content is sent.
- For the sake of transparency, we would like to note: features that would have sent user content to the AI (messaging practice, conversation coach) were planned at one point, but were removed without ever being included in the application because they were incompatible with our privacy principle.

## 6. Data Retention Periods

The fundamental principle of Papatyam's retention approach is that **user control is paramount.** For each data type, either a clear maximum period is defined, or the user has in-app deletion control. KVKK Art.4-d (the principle that data must not be kept longer than necessary for the purpose for which it is processed) is applied together with this user control.

- **Active user data (profile, e-mail, settings, PIN hash, backup code hash, FCM token):** For as long as the user actively uses the application. Purged in cascade when the account is deleted.
- **Secret Hunt sessions and their contents:** 10 days after a Secret Hunt game ends or is cancelled, the conversation content (questions and answers), game information, and all related round records are permanently and irretrievably deleted from the servers. This deletion runs automatically every night at 03:00 Istanbul time via a scheduled task named `bulmacaRetentionKontrol`. Hiding a conversation from your own list does not change this 10-day automatic deletion.
- **Secret Hunt game counter (abuse prevention):** The counter that tracks the number of games opened with the same person is retained until the end of the current month and resets at the beginning of each month. It is only a numeric value; it contains no content information.
- **Template messages:** Retained until the user deletes them via the individual or bulk deletion interface within the application. When the account is deleted, they are purged in cascade on both the sender and recipient sides. No fixed maximum period is applied; the deletion decision rests with the user.
- **Free messages (end-to-end encrypted):** Only the ciphertext (encrypted form) is stored on the server; the server cannot read the plaintext. The message ciphertext is kept until the user deletes it individually or in bulk, or until the account is deleted. **Important note:** The private key that decrypts messages is stored only on your device. If you uninstall the application or your device data is wiped, the ciphertexts on the server can no longer be decrypted — a backup/recovery path is deliberately not designed.
- **Block records (`blocks`):** Records associated with users you have blocked are retained until you remove the block; they are cleared when the block is removed or the account is deleted.
- **Matches and likes:** Retained for as long as the account is active; purged in cascade upon account deletion.
- **Account deletion window (soft delete):** When you delete your account, it is kept in a soft-delete state for 30 days. During this period you can log back in and undo it. After 30 days, the background task `_kullaniciVerisiSilHard` runs and permanently deletes records related to you from all of the following collections: user profile, likes, matches, quotas, block records (bidirectional), template messages (sender and recipient direction), writing-right sessions (both directions), Secret Hunt sessions and sub-round records (both directions), free messages (both directions). A total of 13 collections are cleared.
- **Audit log:** For proof of important actions such as account deletion. Retained for 6 years as required by KVKK Art.7. These records contain only an anonymized hash (derived via HMAC); the user's identity cannot be reversed.
- **Explicit consent records (`kvkkOnayTarihi`, `kvkkPolitikaSurumu`, `kvkkPolitikaURL`):** Retained for as long as the account exists; required as proof of explicit consent (KVKK Art.5/1). When the account is deleted, they are purged in cascade together with other user data; only the pseudonymous audit log record remains.
- **Age confirmation records (`yasOnayi`, `dogumYili`, `yasOnayTarihi`):** Retained for as long as the account exists; required as proof of the 18+ declaration (see 1.5). Purged in cascade when the account is deleted; the pseudonymous audit log record is retained for 6 years (KVKK Art.7).
- **Error reports (Crashlytics):** Subject to the Firebase Crashlytics default retention period (approximately 90 days). Carries the pseudonymous `user_uid_hash` (detail 4.3).
- **System logs (operational):** Technical records kept for error recovery and system health monitoring are retained for at most 30 days. These records contain no conversation content, message text, or decrypted user data.
- **Rate limit records (`rate_limits`):** Automatically deleted with a 48-hour TTL for idempotency. Pseudonymous.
- **Nonces (anti-replay):** Automatically deleted after 10 minutes (TTL policy).

Over all these items, **user control is paramount:** when the account is closed via Settings > Danger Zone > Delete My Account, the soft-delete window begins; for messages, Secret Hunt sessions, and template messages, individual or bulk deletion interfaces are available within the application.

## 8. Your KVKK and GDPR Rights

Under KVKK Art.11 and GDPR Art.15-22, you have the following rights:

### 8.1 Right to Information (Access)
You may request information about your processed data. Most of your data is already viewable within the application (Settings > Profile Information).

### 8.2 Right to Rectification
You may correct your inaccurate or incomplete data through the application (Settings > E-mail, Profile Information).

### 8.3 Right to Erasure ("Right to Be Forgotten")
**You can delete your account from within the application:** Settings > Danger Zone > Delete My Account. After the operation:
- Your account is kept in a soft-delete state for 30 days.
- You can undo it by logging back in within 30 days.
- After 30 days, all your data is permanently deleted in cascade across 13 collections (only the KVKK Art.7 audit log record remains — anonymized, with the user's identity unrecoverable). For details, see item 6.

### 8.4 Right to Data Portability (GDPR Art.20)
You may request that your data be provided in a machine-readable format. For this request, write to sinanfil@yahoo.com.

### 8.5 Right to Object
You may object to data processing. In this case the application may become unusable (since data processing is an essential part of the service).

### 8.6 Automated Decision-Making and Profiling
There is no mechanism in the application that makes automated decisions or profiles users. The matching algorithm works solely through contact phone hash comparison. The artificial intelligence in the application only generates generic questions (see 4.6); it does not analyze or profile users.

### 8.7 Rights Exercise Process
To exercise the rights above:
1. Directly within the application (deletion, correction): immediate effect.
2. For access/portability/objection: write your request to sinanfil@yahoo.com. A response is provided within a maximum of 30 days.

### 8.8 Right to Complain
If our response is not satisfactory:
- Turkey: Personal Data Protection Authority (KVKK) — https://www.kvkk.gov.tr
- EU: Your local data protection authority (e.g., Germany: BfDI, Ireland: DPC).

## 10. Security Measures

- **Encryption:** AES-256-CBC (encrypted profile/e-mail on device), AES-GCM (end-to-end free messages), PBKDF2-HMAC-SHA256 (key derivation), HMAC-SHA256 + pepper (phone and audit hashes).
- **App Check (Play Integrity):** Prevents fake clients from accessing the server (active in production).
- **Replay protection:** Each request is single-use via nonce + timestamp.
- **Rate limiting:** Against spam and brute-force attacks.
- **Device security check:** Warning on rooted/jailbroken devices.
- **Firestore Security Rules:** All write operations are performed via Cloud Functions with the admin SDK. The client cannot write directly.

Despite all these measures, the transmission of data over the internet is not 100% secure; we cannot guarantee absolute security, but we are committed to following best practices.
